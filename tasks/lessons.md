# Lessons — KDP Pipeline

## Image generation

- **1 personnage = 1 prompt** — la génération en grille (4×2, 4×6) produit des personnages mal alignés et de mauvaise qualité. Toujours générer un seul personnage par prompt.
- **Zones noires non coloriables** — si le prompt mentionne des couleurs sombres ("black coat", "dark armor", "shadow"), Gemini remplit ces zones en noir solide. Le résultat est impossible à colorier. Règle : ne jamais mentionner de couleurs dans les prompts de personnages — décrire la forme/texture uniquement (ex: "long coat with detailed patterns" au lieu de "black coat"). Le template dans `pipeline/generate.py` contient déjà "CRITICAL: zero black fills" mais le prompt individuel du personnage doit aussi éviter tout mot de couleur sombre.
- **Format paysage** — Gemini génère souvent en 2816×1536 (paysage). Utiliser `pipeline/clean.py --crop-portrait` pour recadrer. L'algo centre le crop sur le centroïde horizontal des pixels sombres.
- **Artefacts courants** — tête flottante top-left, logo Gemini (étoile 4 branches), texte bas de page. Utiliser `--auto` pour les coins ou `--zones "x1,y1,x2,y2"` pour des zones précises.
- **Cohérence de style** — commencer les prompts avec "Same art style as previous image:" pour enchaîner les personnages avec un style uniforme.
- **StoryForge face swap: scene_reference_images, not preamble** — `gemini-3.1-flash-image` is a generation model, NOT an editing model. Sending the template as `preamble_images` causes the model to reproduce the source image with minimal changes (no real swap). The correct approach: send the template as `scene_reference_images` (labeled "Scene composition reference:") and hero face photos as `reference_images` (labeled "Reference face photos of the target person:"). The model then GENERATES a new image using the template as layout guide and the face photos as hero identity. The prompt must use positional language ("scene composition reference image", "face reference photos") not filenames like `[page_2.png]` which the model cannot resolve.
- **StoryForge face swap: no PIL crop+paste** — Pasting a face crop from the AI-generated image onto the original template creates a visible seam artifact (ghosting circle, style mismatch between realistic original and cartoon-style generated crop). Return the AI-generated image directly — `scene_reference_images` mode already produces a stylistically consistent full image with the hero replaced and the background preserved. PIL blend is counterproductive when source and generated styles differ.
- **StoryForge canonical pages = original illustrations, not AI-regenerated** — The "Render canonical" admin button regenerates template pages using "a generic hero child", producing different illustrations from the real original story art. For boo3-test, canonical pages must be the actual story originals from `images/boo3-test/boo3-test_page_{n+1}.png` (1-indexed originals → 0-indexed canonical `page_{n}.png`). Copy originals into `templates/{slug}/pages/` so the face swap uses the real artwork as scene reference. AI-regenerated canonicals degrade face swap quality because the source scene differs from what customers saw in the store.

## PDF / KDP

- **Jamais reportlab pour l'intérieur KDP** — reportlab injecte une référence Helvetica en boilerplate PDF même sans aucun texte. img2pdf + PIL (texte rendu en pixels) = 0 font ref = 0 erreur KDP.
- **img2pdf produit des PDFs lourds** — 83 MB pour 59 pages est normal (images full res non compressées). La limite KDP est 650 MB, pas de problème.
- **Tester le PDF** — après chaque assemblage, vérifier `Font refs in PDF: 0 [clean]` dans la sortie du script.

## Catégories KDP

- **Ne jamais utiliser "Juvenile Nonfiction"** — ces livres ciblent des adultes millennials. Utiliser : Arts & Photography > Drawing > Manga + Adult Coloring Books.

## Architecture

- **config.py par livre** — chaque livre est autonome. Le pipeline lit n'importe quel config.py sans modification. Ajouter un livre = copier le dossier + éditer config.py.
- **IMAGES_DIR relatif** — utiliser `pathlib.Path(__file__).parent.parent.parent / "images" / "<dossier>"` pour que le projet soit portable.
- **PAGE_SEQUENCE = source de vérité** — l'ordre des pages, les labels, les noms de fichiers. Tout part de là.
- **Story Images vs Coloring Pages**: `pipeline/generate.py` applies a forced B&W lineart conversion and contamination check (`cross_color_contamination` > `strip_colors`). For a `story` category, it is critical to **bypass** these cleanups so the final AI-generated children's book illustrations remain fully colored.
- **Dynamic Prompts**: `pipeline/prompt.py` has a rigid global `_BASE_TEMPLATE` enforcing "PURE BLACK AND WHITE". A separate `build_story_prompt` template must be used for stories to ensure artistic harmony and proper colors.
- **Lost Generations (Only 1st Page)**: This occurs when the `config.py` did not successfully synchronize the `PAGES` array from the dashboard before `generate.py` ran. A resilient dashboard saves (`PUT` vs `POST` routing fix) ensures all pages are written, so the backend loop sees everything.
- **UI Overflow**: Wrapping HTML structures that employ Tailwind's `overflow-y-auto flex-1` must be meticulously checked for rogue `</div>` closures, which can break scrolling and truncate the bottom contents of a layout.
- **Granular Generation**: Passed specific IDs (page numbers) straight to generative endpoints (`generate.py --id <item>`) to provide item-level regeneration instead of batch-ony runs.
- **PDF Layout Modification**: Text logic was extracted to `pipeline/draw_utils.py` handling exact font width checking so lines break correctly (`draw_text_wrapped`) across varying font lengths when dealing with multilingual text on storybook images.
- **Introduction and Outro Pages**: New functions `make_intro_page` and `make_values_page` assemble fully autonomous custom pages using `img2pdf` compatibility formatting while strictly abiding by the 0-font insertion rule by utilizing PIL directly on blank canvas masks.
- **CSS Grid/Flex Heights**: When combining `h-full` and `flex-col`, make sure parent containers explicitly use `overflow-hidden` otherwise inner elements with `flex-1 overflow-y-auto` will expand the parent layout rather than scroll, causing half-visibility or cut-off interfaces.
- **Story Layout Varieties**: Introduced discrete rendering loops (`make_text_only_page`, `make_overlay_story_page`, `make_story_page`) to support multiple presentation forms while preserving KDP formatting rules.
- **Language Iteration Output**: Instead of returning a single `FINAL.pdf`, iteration loops wrap the PDF assembly process producing `<book>_<lang>_interior_FINAL.pdf` output matrices corresponding to configured UI checkboxes.

## Dashboard UI reliability

- **Apply + Regenerate must fail loudly** — in prompt-builder actions, never swallow save/network errors before generation. Show explicit API error details (`detail`) so users know why generation did not start.
- **Builder selections must persist per character** — temporary UI selections (style/pose/elements/theme/notes) should be cached by `book + character id` (localStorage) so re-render and auto-reload do not wipe the visible selected state.
- **True regenerate requires `force=true`** — without force, generate endpoint skips existing files and users perceive it as "nothing happened". For per-image regenerate actions always pass `force=true`.
- **Avoid full-page reload for single-image flows** — SSE completion should support non-reload mode with an `onDone` callback to cache-bust only affected `<img>` nodes and keep user context (selected character + builder state).
- **Tag category semantics matter** — style/pose/elements/theme are single-choice per line for this product. Toggle logic must replace previous choice in category, not append multi-select arrays.
- **Always clear list container before rerender** — in dynamic panels like `PAGE_SEQUENCE`, forgetting `list.innerHTML = ''` causes duplicate stale rows and hides fresh image updates, making regeneration look broken.
- **Serve regenerated images with anti-cache headers** — image endpoints must send `Cache-Control: no-store` (plus pragma/expires) or browsers may keep stale PNGs even after reload.
- **Version image URLs on render** — append a cache-bust query (`?v=<timestamp>`) to dynamic image URLs and bump it after regeneration to guarantee fresh bytes in the current UI session.
- **Do not gate UI refresh on perfect logs** — regeneration post-refresh callback should run on stream completion (`[DONE]`) even when warnings/errors are printed, otherwise users see stale previews despite updated files on disk.
- **Normalize sequence filename variants to character IDs** — `PAGE_SEQUENCE` files like `_portrait`/`_clean` do not equal real character IDs. Regenerate and preview lookup must normalize suffixes and resolve against `CHARACTERS`, otherwise generation targets wrong IDs and refresh appears broken.
- **Preview should follow canonical generated file, not sequence variant** — for character dashboards, render image previews from `bookPrefix_<charId>.png`; keep `PAGE_SEQUENCE` filename only as metadata. This avoids showing wrong legacy `_portrait` outputs.
- **Generation endpoint must resolve key from dashboard config too** — relying only on `os.environ` for `GEMINI_API_KEY` breaks regeneration even when key is saved in `settings.json`/`.env.local`.
- **Portrait crop must be content-aware** — fixed center-strip crop can cut the subject; use dark-stroke weighted horizontal centroid to keep the character in frame when converting landscape to portrait.
- **Do exact ID match before normalization** — stripping suffixes like `_portrait` can break valid character IDs (e.g. `vegeta_portrait`) and send regeneration to wrong targets.
- **Never auto-rewrite PAGE_SEQUENCE filenames during regenerate** — forced canonicalization can disconnect curated sequence assets from character IDs; resolve mapping for actions, but preserve sequence filenames unless user explicitly edits them.
- **Anchor builder prompts to character identity** — prepend immutable identity (`name,id,series`) so tag-only edits do not drift to generic non-character outputs (e.g. Naruto looking unrelated).
- **DOM-only cache bust is not enough for reliable instant refresh** — after regeneration, run an in-place data refresh (`loadConfig` + reselect character) and then image cache-bust; this avoids stale UI state without requiring full page reload.
- **Reference fidelity needs explicit schema fields** — storing only free-text prompt is fragile; add optional structured metadata (`source_type`, `source_title`, `source_character_name`) in `CHARACTERS` and inject it into solo/group prompt context to preserve character/source consistency.
- **Reference inputs must drive live prompt composition** — source fields should trigger preview updates on input and prompt builder should read live form values (not only cached character objects), otherwise users think reference data is ignored.
- **Never store fully templated prompts inside `CHARACTERS.prompt`** — dashboard should save compact character description only. `pipeline/generate.py` already applies the global template via `build_prompt`; storing a full template causes double-wrapping and weakens identity/reference adherence.
- **Harden generation against legacy prompt debt** — `build_prompt` must sanitize/compact stale nested prompt strings (strip wrappers + dedupe repeated segments) so existing configs still generate correct character-anchored outputs.
- **Sanitize at config boundary, not only at generation** — normalize `CHARACTERS.prompt` during config read/write so legacy nested prompts are permanently cleaned; otherwise users can regenerate repeatedly with effectively unchanged compact core and perceive "no change".
- **Sanitize prompt-builder local state on load** — cached `soloState.description` can reintroduce legacy wrapped prompts in preview/save flows even after backend fixes. Always compact description when hydrating from character prompt and localStorage.
- **Prompt constraints are not enough for line-art quality** — add a deterministic final binary pass after fill cleanup so exported coloring pages contain only pure black outlines and white interiors (no gradients/gray shading).
- **Deep fill removal alone misses medium black islands** — add a one-pass micro-core hollowing step (single erosion core whitening) before final binarization to clear stubborn solid patches (hair spikes/accessory fills) while keeping thin outlines.
- **Avoid global hard binarization in default pipeline** — strict 0/255 conversion can create jagged lines and missing details. Use quality-preserving cleanup by default: color strip + dense-component fill removal + gray shading cleanup, with strict binarization only as optional emergency mode.
- **SSE streams need disconnect-safe termination** — browser `ERR_INCOMPLETE_CHUNKED_ENCODING` can occur on interrupted EventSource streams; backend must guard subprocess lifecycle/decoding and suppress trailing write errors, while frontend should ignore post-`[DONE]` transport errors and still run final callbacks on premature disconnect.
- **Avoid orphan route decorators in FastAPI** — a stray `@app.get("/stream/assemble/{book_name}")` above another handler can silently bind assemble requests to wrong function (e.g., cover generation), making selected output folders appear ignored even though UI flow succeeds.
- **Normalize uploaded hero photos to real PNG bytes** — when `/api/storyforge/{name}/photos` stores raw JPG/HEIC bytes with a `.png` filename, `build_hero` later sends invalid `image/png` references to Gemini and gets `400 INVALID_ARGUMENT`. Always decode uploads with PIL and re-encode as PNG before saving `source_*.png`.
- **Multilingual page text must follow selected languages, not a hardcoded dict** — StoryForge originally wrote `{"fr": text, "ar": "", "en": text, "es": ""}` with `languages: ["fr"]`, so English template text leaked into both fr+en, ar/es were empty, and the config claimed French only. Fix: `translate_pages(specs, source_lang, selected_langs, translate_fn)` keeps the template's `language_default` text as source and fills the other selected languages via Gemini; `build_book(..., languages=, page_texts=)` writes exactly the chosen languages. UI language checkboxes must align with what is generated.
- **Page count must change book length, not just price** — the StoryForge page-count selector wires live expansion: when `page_count != len(template.pages)`, the generate route calls `expand_narrative(...)` (Gemini text model returns exactly N `{text, image_prompt}` pages) instead of `resolve(...)`. Validate the model returns exactly the requested count and preserve all `{TOKEN}`/`{HERO}` braces verbatim.
- **Pricing is admin-configurable, never hardcoded in UI** — `pipeline/pricing.compute_price` is pure and reads a `pricing` block from `settings.json` (placeholder values: per-page B&W/color, paper-quality multiplier, cover cost, markup). The dashboard fetches `/api/pricing` on every option change for a live estimate. Unknown paper quality → `ValueError` → 400.
- **Cover image: hero only, title added later** — `storyforge.generate_cover(title, hero, image_gen)` never puts the title text in the AI prompt (matches the "never embed title in AI cover" rule); it passes the canonical portrait as a reference image and the title is overlaid programmatically afterward.
- **Storefront vs admin are separate surfaces** — `/book/{name}` stays admin (full config editor + Generate/Clean/PDF/Cover/Publish). Customers use `/store` (published catalog), `/store/{slug}` (linear personalize → email-code auth → checkout). Only books with `PUBLISHED = True` appear in the catalog or are reachable at `/store/{slug}` (404 otherwise).
- **Email-code auth uses stdlib only** — no `itsdangerous`/`stripe`. Codes are hmac-sha256 hashed with a per-code salt, TTL 600s, rate-limited at 5 attempts (`storefront/auth.py`). Sessions are HMAC-signed base64url cookies with an `_iat` timestamp (`storefront/session.py`), set httponly + samesite=lax. Payment is a `StubPaymentProvider` returning a pending session pointing at an internal mock URL; swap via `get_payment_provider(settings)` later.
- **Shared TestClient persists cookies** — Starlette's `TestClient` keeps cookies across requests, so a prior auth test leaks `sf_session` into later tests. Call `client.cookies.clear()` at the start of the storefront fixture or the unauthenticated 401 cases pass through as 200.
- **Orders + admin live in one SQLite file** — `storefront/db.py` (`Database`) holds `orders`, `auth_codes`, `admins` with idempotent `CREATE TABLE IF NOT EXISTS` migrations, stdlib `sqlite3` only (one connection + `threading.Lock`, `check_same_thread=False`). The JSON `AuthStore` is superseded by `SqliteAuthStore(db)` in the app; the app caches the db in `_SF_DB_HOLDER` and seeds admins from settings on first open.
- **Customer order flow is two steps** — `POST /store/{slug}/order` (multipart child_name+photo, auth-gated) re-encodes the photo to PNG, saves it under `.storefront/orders/{reference}/photo.png`, prices via `compute_price`, and inserts a `pending` order. `POST /store/{slug}/checkout` then takes `{reference}`, verifies ownership by session email, and the stub marks it `paid` immediately. Tests that hit `/store/{slug}/order` write into the gitignored `.storefront/`; `rm -rf .storefront` between runs to avoid accumulation.
- **Admin auth is opt-in for the VPS** — `storefront.admin.enabled` defaults false so local dev stays open; when true, `_require_admin` gates `/`, `/book`, `/settings`, `/storybook` (redirect to `/admin/login`) via a separate signed `sf_admin` cookie. Only emails in `storefront.admin.emails` (seeded into the `admins` table) can request a code. Set `storefront.https: true` on the VPS so auth cookies get `secure=true`.
- **Portable secret bootstrap beats manual VPS setup** — For `storefront` auth/session signing, avoid hardcoded insecure defaults and avoid requiring manual one-off secret generation on server login. Use precedence: `settings.json storefront.session_secret` -> env (`STOREFRONT_SESSION_SECRET`, fallback `SESSION_SECRET`) -> persisted auto-generated file `.storefront/session_secret.txt`. This keeps single-server deployment zero-touch while allowing explicit shared secret in multi-instance production.
- **Auto-release ports on relaunch** — standard `Makefile` launches or python script server re-runs can crash due to `Address already in use` (port 8000) when orphaned child processes (e.g. from uvicorn reload or multiprocessing) hang. Adding a pre-launch check using `lsof -t -i:8000` to find and kill conflicting PIDs automatically avoids manual cleanup and ensures robust server startup.
